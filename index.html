Beekeeper

beekeeper is the control tower for Bee test clusters. It creates clusters and runs health checks so we can quickly see if the network is working.

Top things beekeeper can do

Create and delete Bee clusters from config profiles.
Run multiple check types for connectivity, sync, retrieval, postage, settlements, and stake/withdraw flows.
Operate nodes with restart, nuke, print, and node-operator workflows.
Export check telemetry to Prometheus pushgateway and visualize in Grafana.

1) Command entry and config loading

Startup path

  • Root command beekeeper initializes all subcommands.
  • Persistent pre-run calls config init before command execution.
  • Global flags include kube, tracing, loki, geth RPC, config source.
  • Optional periodic execution wrapper for continuous operations.

Example full command

  • Use this to create an autotls local cluster with verbose logs:
  • ./dist/beekeeper create bee-cluster --cluster-name=local-dns-autotls --log-verbosity=debug

2) Cluster setup lifecycle

step 1: validate cluster name and config profile lookup.
step 2: if starting a new cluster, require --geth-url and --wallet-key.
step 3: setup bootnode groups first, gather addresses, then fund them.
step 4: setup remaining node groups (supports static endpoints or deploy/add).
step 5: fund non-bootnode addresses and complete setup.

Implementation: goroutine fan-out per node setup, channel aggregation for errors and ETH addresses.

3) Top commands you will use most

Daily operations commands

  • create bee-cluster: start cluster from config profile.
  • delete bee-cluster: remove cluster resources.
  • restart: restart selected nodes safely.
  • nuke: clean up network state when needed.
  • print: inspect cluster/node information quickly.

Funding and verification commands

  • fund: fund cluster wallets for ETH/BZZ needs.
  • node-funder: periodic top-up by address/namespace/cluster.
  • stamper: create/topup/dilute/set postage batches.
  • check: run integration checks on cluster behavior.
  • node-operator: operational node actions in k8s.

These are wired in cmd/beekeeper/cmd and share global flags (k8s, tracing, metrics, geth RPC, config source).

4) Bee ci workflow when a PR is open

How it works in practice

  • For Bee PRs, CI runs workflows that include beekeeper-based integration checks.
  • A common path is the Bee workflow file .github/workflows/beekeeper.yml.
  • Checks validate cluster behavior (connectivity/retrieval/autotls and related paths).
  • On failures, workflow can collect debug artifacts for investigation.

Example

  • PR ethersphere/bee#5350 adds AutoTLS integration coverage in beekeeper workflow.
  • It extends CI flow with a dedicated AutoTLS cluster test path and artifact handling.
  • Use this as a model when explaining why a beekeeper check change matters for Bee CI.

5) How we use beekeeper day to day

Where it runs

  • local dev: with beelocal environments (k3d/k3s style local clusters) for quick iteration.
  • ci: Bee PR workflows run beekeeper checks to catch regressions before merge.
  • shared k8s clusters: used across environments like mainnet/public-testnet and bee-light-testnet dev clusters.
  • namespace mode: can target by cluster-name or directly by namespace + label selectors.

How it is operated

  • interactive runs: operators trigger one-off checks and restarts from CLI.
  • job mode: deploy beekeeper as a Kubernetes job for single-run checks.
  • periodic mode: run as cronjob / periodic command execution for recurring validations and funding checks.
  • observability loop: send metrics to pushgateway and inspect results in Grafana dashboards.

6) Metrics and dashboard visibility

What metrics path exists in code

  • check command can enable metrics and push to pushgateway.
  • Check actions and logger can register collectors through metrics middleware.
  • This gives run-level visibility for failures, durations, and behavior over time.

Grafana example

  • Smoke check performance dashboard for beekeeper runs:
  • beekeeper smoke check performance
  • Use it to spot regressions quickly after PR checks.

7) Takeaway

  • what it is: beekeeper is the operations cockpit for Bee test clusters, combining lifecycle actions and validation checks in one toolchain.
  • most important flow: choose config source (local or git), create or attach to cluster, run checks, and read outcomes in CI + metrics dashboards.
  • commands that matter most: check, restart, fund, stamper, print, and nuke.
  • operational guardrails: for cluster creation always provide --geth-url and --wallet-key; for check configs only override values that differ between scenarios.
  • why metrics are key: pushgateway + Grafana make regressions visible quickly and help compare behavior across PR runs.
